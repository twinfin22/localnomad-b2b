# 2차 시장 조사 vs Phase 1 Spec v2.0 상세 비교분석

**작성일**: 2026년 2월 13일
**대상 리서치**: 10개 주제 (IEQAS, HireVisa, Legal, FIMS, Kakao, Target Universities, AI Multilingual, GTM, Competitors, Financial)
**비교 대상**: Phase 1 Spec v2.0

## 1. IEQAS 인증제도 분석

### 조사에서 발견된 주요 사실
- **2024년 인증 현황**: 학위과정 158개교, 어학연수 103개교 (전년도 대비 각각 +24개, +13개)
- **2025년 우수인증**: 39개교 선정 (선문대 13년 연속, 호서대 12년 연속)
- **기본 기준**: 불법체류율 2% 미만 (우수 1% 미만)
- **2025년 2학기부터 24개 대학 비자 발급 제한**: 세한대, 초당대, 광주여대 등
- **비수도권 유학생 비중**: 45.8% (상승추세)
- **전체 불법체류율**: 약 17.64% (2023년 7월 기준)
- **정부 목표**: Study Korea 300K (2027년까지 30만명)
- **IEQAS 평가 기한**: 연 4회 (2월, 5월, 8월, 11월)

### v2.0에 반영된 내용
- Phase 1 목표에서 "IEQAS 불법체류율 추적"을 핵심 차별화 포인트로 명시
- 트래픽 라이트 상태 엔진에 "IEQAS 불법체류율 현황: 0.3% (목표 1% 미만)" 대시보드 카드 추가
- 사용자 페르소나 P3 (국제교류처 과장)의 핵심 기대: "IEQAS 실시간 불법체류율 지표"
- 선문대(13년 연속), 호서대(12년 연속) 우수인증 사례를 아산권 3개 대학 타겟의 근거로 명시
- 타겟 대학 선정 기준에 "IEQAS 인증 보유 또는 강한 관리체계"를 포함

### v2.0에 미반영된 내용 (중요한 누락사항)
1. **IEQAS 평가 기한 자동 알림 시스템 부재**
   - 연 4회(2월, 5월, 8월, 11월 말) 평가 기한과 동기화한 자동 알림 로직이 없음
   - F1-12 알림 시스템에는 "FIMS 정기보고 마감"은 있으나 IEQAS 평가 기한은 빠짐

2. **비자 제한 위험 대학 모니터링 기능 부재**
   - 24개 제한 대학을 추적하고 제한 기간 중 맞춤형 관리 지원 기능 미명시
   - 로컬노마드가 "규제 회피 솔루션"으로 포지셔닝할 수 있는 기회 미활용

3. **불법체류 "고위험 임계값" 세부 정의 부족**
   - 연구에서: "기본인증 2% 미만, 우수인증 1% 미만"
   - Spec에서: 트래픽 라이트 엔진은 개별 학생 체류 상태 기반만 있고, 전체 대학 불법체류율 목표값을 어떻게 관리할지 미명시
   - 만약 현재 0.3%에서 0.8%로 상승하면 어떤 액션을 취할지 워크플로우 부재

4. **IEQAS 인증 유지/탈락 시나리오별 기능**
   - 인증 탈락 위험 대학용 특화 리포트 또는 경고 시스템 미명시
   - 예: "현재 추세 지속 시 다음 평가에서 우수인증 탈락 예상" 같은 예측 분석 기능 부재

### 반영률 평가
- **반영률**: 약 40% (기본 개념은 반영되었으나 실행 레벨 기능이 부족)
- **근거**: 불법체류율 추적이라는 아이디어는 들어갔으나, IEQAS 평가 주기와 연동한 체계적 관리는 미흡

## 2. HireVisa(하이어비자) 경쟁사 분석

### 조사에서 발견된 주요 사실
- **시장 점유율**: 서울 거주 외국인 유학생의 81% 이상 사용
- **협약 대학**: 105개 이상 (서울대, 연세대, 고려대, 홍익대 등)
- **투자 규모**: 시드 10억(2024년 초) → 시리즈A 50억(2025년) = 누적 60억원
- **직원 규모**: 약 19명
- **핵심 경쟁력**: 대학과의 파트너십 + B2B2C 모델 + 학교 대시보드 제공
- **주요 약점**:
  - 개인화된 경험 부족
  - 주거(Housing) 솔루션 미지원
  - 공개 가격 정보 없음
- **금융 서비스 확장**: 하나은행 제휴로 외국인 금융서비스 진출
- **확장 계획**: 2024년 5월 전문대학 서비스 출시

### v2.0에 반영된 내용
- Phase 1 목표 도입부에서 "하이어비자(HireDiversity, 105개 대학 협약, 60억 투자)가 제공하는 핵심 기능을 모던 UX로 재구현"이라고 명시
- 로컬노마드의 차별화 전략:
  - "IEQAS 불법체류율 실시간 추적" (하이어비자가 못함)
  - "AI 자동화"
  - "FIMS 자동화"
  - "다국어 AI 상담" (하이어비자 불만족 지점)
- 경쟁 관계 이해: "대학 주도 B2B 모델 vs 학생 주도 B2C 모델" 포지셔닝
- 가격 전략: 재무 모델에서 하이어비자 추정 가격 대비 30-50% 저가 전략 명시

### v2.0에 미반영된 내용 (중요한 누락사항)
1. **하이어비자의 "단체 접수" 프로세스 상세 분석 부재**
   - 연구: "현금 50,000원, 오프라인 제출, 지정된 시간/장소에 방문" → 10배 편리한 "완전한 비대면"으로 해결
   - Spec F1-10 "단체 접수 데이터 일괄 준비"는 구비 서류 체크리스트만 있고, 실제 "완전 비대면 온라인 단체 접수"는 Phase 1 스코프 외로 표시됨
   - 이는 하이어비자 대비 가장 강력한 차별화 포인트인데 Phase 1에서 미제공

2. **하이어비자 금융 서비스 확장에 대한 대응 전략 부재**
   - 연구: "하나은행 제휴로 금융서비스 확대 추진"
   - Spec에는 금융 서비스 통합 관련 언급 전무
   - "부동산 중개"와 "생활 정착 컨시어지" 같은 B2B2C 확장 전략도 Phase 1에 미반영

3. **직원 19명, 빠른 성장에 따른 조직 확대 필요** 관점의 기회 분석 부재
   - 하이어비자가 19명으로는 105개 대학 풀서비스 불가능 → 로컬노마드의 "AI 에이전트 4-5x 생산성" 강점을 더 명시적으로 활용 가능
   - Spec에는 AI 에이전트 생산성 명시는 있으나, 경쟁사 약점과의 연결고리가 약함

4. **하이어비자의 "정보 공백" 고통점** 미활용
   - 연구: "실물 카드가 나오기 전까지 등록번호를 모르는 정보 공백" → "패스트트랙 번호 조회" 제안
   - Spec에 이 기능 없음 (아마도 이미 충분하다고 판단한 듯)

### 반영률 평가
- **반영률**: 약 60% (전략적 포지셔닝은 좋으나, 운영 레벨 차별화 기능 부분 부족)
- **근거**: 경쟁 구도는 이해했으나, "10배 편리함" 구체적 구현(비대면 단체 접수, 부동산 서비스 등)은 Phase 2 이후로 미루어짐

## 3. 법적 포지셔닝 분석

### 조사에서 발견된 주요 사실
- **삼쩜삼 판례의 핵심**: "도구 제공" vs "전문자격 행위"의 경계
  - SaaS: 도구 제공 → 합법
  - 행정사 행위: 행정사 도장 날인, 명의 대리 → 자격 필수
- **행정사법 제2조**: 행정사는 행정기관 제출 서류 작성/제출 대행 가능
- **비자 관련 행정사 업무**:
  - 가능: 비자발급신청서 작성 및 제출 대행
  - 불가: 해외 공관 비자 신청 대행, 법적 판단 제시
- **개인정보보호법 제26조**: 대학이 SaaS 제공자에게 위탁 시 위탁계약 필수
  - 필수 항목: 위탁 업무 범위 한정, 기술적/관리적 보호조치, 재위탁 제한, 감독 권리, 손해배상 책임
- **ISMS-P 인증 비용**: 간편형 600만~1,100만원 (소요기간 6개월+)
- **AI 챗봇의 법적 한계**:
  - "정보 제공" → 안전 (예: 비자 종류, 절차 안내)
  - "법적 의견" → 위험 (예: "당신의 경우 이 비자가 맞습니다")

### v2.0에 반영된 내용
- Feature 2 (AI 봇)의 "Out of Scope" 명시:
  - "개별 사안 법적 판단" → 에스컬레이션
  - "허가 가능성 판단" → 에스컬레이션
  - "불법체류 관련" → 긴급 에스컬레이션
  - "개인정보 요청" → 거부
- "도구 제공" 원칙 명시: "F1-01 학생 프로필", "F1-04 엑셀 임포트 + AI 컬럼 매핑" 등은 모두 도구 차원
- 개인정보보호 고려: 학생 데이터 필드에서 여권번호, 외국인등록번호 AES-256 암호화 저장 명시
- AI 봇 면책 문구: "⚠️ AI 번역 참고용입니다. 정확한 정보는 국제교류처에 문의하세요" 자동 추가 (우즈벡/몽골어)
- RAG 기반 "정보 제공만 하는" 설계: 출입국관리법, FIMS 가이드라인 등 공개 정보 기반

### v2.0에 미반영된 내용 (중요한 누락사항)
1. **ISMS-P 간편인증 취득 로드맵 부재**
   - 연구: "초기 6개월 내 간편인증 취득 목표 (약 900만원)"
   - 재무 모델: "ISMS-P 간편인증 900만원" 명시
   - Phase 1 Spec: "ISMS-P 간편인증 (별도 프로세스)"라고만 표시, 실제 W0~W6 일정에 언제 신청할지 미명시
   - 실제로는 ISMS-P 심사 기간이 2개월 이상이므로, W0부터 신청해야 Phase 1 완료 후 인증 가능 → 타임라인 문제

2. **행정사 네트워크 구축 전략 부재**
   - 연구: "행정사와의 파트너십 모델이 안전" + "행정사 네트워크 구축하여 최종 검토 담당 고려"
   - Spec: 로컬노마드가 모든 행정 행위를 피하는 도구 수준으로만 설계
   - 하지만 현실적으로 대학은 여전히 "누가 책임지는가"를 묻는데, 행정사 파트너 명시가 신뢰도를 높일 수 있음 → 비즈니스 모델 고려 부족

3. **위탁계약(계약서) 템플릿 준비 일정 명시 부재**
   - 연구: "개인정보보호법 제26조 위탁계약이 필수"
   - 재무 모델: "계약서 템플릿 200만원" 명시
   - Phase 1 Spec W0~W6 일정에는 "법무 자문" 항목이 없음 (법무는 W0에 사전 조사만 가정)
   - 실제로 파일럿 대학과 계약 시 위탁계약 필수인데, 준비 일정이 불명확

4. **국외 데이터 이전(AWS Seoul vs Global)의 구체적 선택 미명시**
   - 연구: "한국 내 데이터센터 사용 권장" + "AWS Seoul 리전 이용이 합법성 강화"
   - Phase 1 기술 스택: "AWS Seoul 인프라 셋업"은 명시되었으나, 클라우드 서버 사용이 ISMS-P 위탁 판정의 근거가 되는지에 대한 법적 검토 결과 미제시
   - AI 에이전트(Claude API)의 데이터 처리 위치는? 미명시

5. **"금품을 받고 법률상담"(변호사법 제109조)과의 경계**
   - AI 챗봇이 유료 서비스(Basic 3,000원/학생/월)인데, 법적 정보 제공이 "법률상담"으로 해석될 수 있는지 법적 검토 부재
   - 로컬노마드가 SaaS 요금을 받으므로, "금품을 받고"에 해당할 가능성 → 법적 자문 필요

### 반영률 평가
- **반영률**: 약 50% (법적 원칙은 이해했으나, 구체적 실행 계획과 법적 위험 완화 단계 부족)
- **근거**: 도구 vs 행위 구분은 좋으나, ISMS-P 일정, 위탁계약, 데이터 처리 위치, 변호사법 경계 등에서 구체성 부족

## 4. FIMS(유학생정보시스템) 분석

### 조사에서 발견된 주요 사실
- **FIMS 접속 권한**: 대학 체류관리담당자만 가능 (로컬노마드는 직접 접근 불가)
- **정기보고 주기**: 연 4회 (2월, 5월, 8월, 11월 말)
- **변동신고 기한**: 사유 발생 후 15일 이내
- **수집 데이터**: 학생명, 국적, 학번, 외국인등록번호, 학과, 성적, 출석률, 체류자격, 체류기간 등 13종
- **현재 문제점**:
  - 시스템 사용성 낮음 (복잡한 입력 절차)
  - 대학 학사시스템과의 수동 입력 방식 (자동화 부재)
  - 정기보고 기한 준수 부담 높음
- **로컬노마드의 기회**: 대학 학사시스템 데이터를 FIMS 형식으로 자동 변환/검증 기능으로 대학의 운영 부담 70% 이상 감소 가능

### v2.0에 반영된 내용
- **F1-07 FIMS 양식 호환 내보내기**:
  - 내보내기 유형: 학생 정보, 변동신고, 정기보고 통계, 단체 접수 데이터, 범용 내보내기
  - 미리보기 → 누락 필드 경고 → 다운로드 워크플로우 구현
- **F1-08 FIMS 변동신고 워크플로우**:
  - 15일 카운트다운 표시
  - 기한 7/3/1일 전 단계별 알림
  - 신고 완료/미완료 체크리스트 관리
  - 변동신고 이력 전체 조회
- **F1-12 알림 시스템에 "FIMS 변동신고 기한"과 "FIMS 정기보고 마감"** 명시
- **학생 프로필 데이터 필드**: FIMS 13종 데이터 필드 대부분 포함 (학번, 국적, 학과, 출석률, 학점, 비자유형 등)
- **W0에서 파일럿 대학으로부터 FIMS 양식 확보** 명시 (전제조건)

### v2.0에 미반영된 내용 (중요한 누락사항)
1. **"대학 학사시스템과의 자동 연동" 전략 부재**
   - 연구: "대학 학사시스템(SAP, 신라정보 등)과 연동하여 자동 수집 가능"
   - Spec F1-04 엑셀 임포트는 있으나, "학사시스템 → FIMS" 자동 파이프라인은 Phase 1 스코프 외
   - 호서대, 선문대, 순천향대가 어떤 학사시스템을 쓰는지 조사 부재
   - Phase 2/3로 미루어짐: "학사 시스템 직접 API 연동 (Phase 2)"

2. **FIMS 서버 장애 시 대응 전략 부재**
   - 연구: "FIMS 인프라 유지보수로 인한 서비스 중단 발생"이라는 현황 명시
   - Spec에는 오프라인 모드나 로컬 캐시 같은 FIMS 미연결 상황 대응 로직 없음
   - 만약 FIMS 마감 3일 전에 FIMS 서버가 다운되면 로컬노마드도 도움이 못 됨

3. **"FIMS 직접 API 연동 불가"의 명확한 표시 필요**
   - 연구: "FIMS 접속권한 없음 (정부 보안상 대학만 접근)" → "직접 API 연동 불가"
   - Spec F1-07/F1-08은 마치 FIMS에 자동으로 데이터가 들어가는 것처럼 보임
   - 실제로는 "대학 담당자가 수동으로 복사-붙여넣기" 단계가 필수인데, 이 경계가 명확히 표시되지 않음
   - 파일럿 대학 담당자가 "자동으로 FIMS에 입력되는 건가?" 착각할 수 있는 위험

4. **정기보고 통계 "교육과정별/사유별 집계표" 구체 미정**
   - 연구: "정기보고: 교육과정별(학사/석사/박사/어학연수) + 유학중단사유별(학업부진/경제적 어려움 등) 집계"
   - Spec F1-09 "정기 현황 보고서 자동 생성"은 있으나, 구체적 집계 로직이 명시 안 됨
   - FIMS 양식에 맞는 정확한 분류 기준이 있어야 함 → W0에서 파일럿 대학 FIMS 양식과 함께 수집 필요

5. **FIMS 변동신고 "트리거 조건" 한정성**
   - 연구에서 감지된 변동: "휴학/제적/자퇴/졸업/미등록/소속변경"
   - Spec F1-08: 위 6가지 모두 포함됨
   - 하지만 "출석률 급감(예: 80%→30%) 후 복구 안 됨"이나 "성적 F 연속" 같은 간접 지표는 없음
   - 대학별로 자체 규정(예: "평점 2.0 미만 제적")이 다르면? 커스터마이징 가능성 미제시

### 반영률 평가
- **반영률**: 약 70% (FIMS 양식 호환 기능은 잘 구현되었으나, 학사시스템 통합과 자동화 로드맵이 모호)
- **근거**: F1-07/F1-08/F1-10은 구체적이고 좋으나, "10배 편리함"을 약속하려면 학사시스템 자동 연동이 필수 → Phase 2로 미루는 것이 맞지만, 그 일정이 불명확

- **즉시 배포 가능 언어**: 한국어, 영어, 중문, 베트남어
- **조건부 배포 언어**: 우즈벡어, 몽골어 (전문가 검수 후 제한적 배포)

### v2.0에 반영된 내용
- **초기 투자 비용**: 재무 모델에서 상세히 제시된 8,000만원을 Spec에서는 언급 안 함 (당연히 필요한 것으로 간주)
- **W0 사전 준비**: 사업자등록증, ISMS-P 신청, KAFSA 가입, 파일럿 대학 사전 연락 등이 필요하지만, Spec에는 "W0 (Pre-sprint)"이라고만 표시
- **M3 분기점**: Phase 1이 W6 = 약 6주 = 1.5개월인데, 재무 모델 M1~M3는 3개월이므로 시간 스케일 불일치
- **가격 모델**: Basic 3,000원/학생/월 → 호서대(1,200명) = 월 360만원 = 연 4,320만원
  - 하지만 재무 모델 M1 첫 대학은 "800명 Basic = 월 240만원"이므로, 호서대가 아닌 작은 대학부터 시작하는 것으로 가정?

### v2.0에 미반영된 내용 (중요한 누락사항)
1. **"W0~W6" 과정에서의 현금흐름** 미명시
   - Spec: "W6 완료 후"에 MVP가 어느 정도 완성되는가?
   - 재무 모델: M1부터 매출이 들어온다고 가정 (실제로는 W6 후 M1부터 매출?)
   - 시간 스케일 혼동: "W0~W6 = 6주 + Pre-sprint"인데, 이게 M1~M6의 어느 구간인가?

2. **"ISMS-P 인증 없이 매출" 가능 여부** 미명시
   - 재무 모델: "ISMS-P 간편인증 900만원" 초기 투자로 명시
   - Spec: W0에서 신청하면 M3~M4쯤 완료 가능한데, 그 사이에 매출이 들어오면?
   - 현실: 대학과 계약 시 "ISMS-P 인증이 없으면 계약 불가"일 가능성 높음 → M3 전에 인증 필수

3. **"초기 3개 대학" 선정의 현실성** 검증 부재
   - 재무 모델: M1~M3에 아산권 3개 대학 온보딩
   - Spec: "파일럿" 기간 중 W0~W6 + 추가 계약 협상?
   - W6가 끝나고 M1부터 매출이 들어오려면, 계약 서명이 W5 또는 W6에 되어야 함 → W0에서 사전 협상 필수인데, 명시 안 됨

4. **"변동비" 계산의 부정확성**
   - 재무 모델: "Claude API: 학생 1,000명당 월 50만원" (1개월 50건 쿼리 기준)
   - Spec F2: "응답 시간 5초 이내" → 실제 API 호출 비용은?
   - 카카오 알림톡: "월 20~100만원 추정" → 정확한 발송 비용 미파악 (W0 조사 필요)

5. **"M12 후" 확장 계획** 불명
   - 재무 모델: 기본 시나리오 M12에 월 4,050만원 (연 48,600만원 ARR)
   - Phase 2/3 계획은? ARR 1억원/2억원 달성 언제?
   - Series A 펀딩 후 영업팀 확대는? 인력 비용은?

6. **"보수적 시나리오의 위험" 완화 전략** 미제시
   - 재무 모델: M8-M9에 현금 부족 위험
   - Spec: 이 위험에 대한 대응책이 없음 (엔젤 투자 조기 신청? 가격 인상? 다른 수익 원?)
   - W0~W6 중에 "리스크 미티게이션" 계획 필요

### 반영률 평가
- **반영률**: 약 45% (재무 개념은 포함되었으나, 타임라인 동기화와 구체적 위험 완화 전략이 부족)
- **근거**: 초기 투자 8,000만원과 기본 시나리오 월 4,050만원은 Spec에 영향을 주었으나, W0~W6 타임라인과 M1~M12 재무 계획이 정렬되지 않음

## 결론

Phase 1 Spec v2.0은 **전략적 포지셔닝(HireVisa 대비 차별화, B2B 타겟, 가격 경쟁력)은 우수**하나, **운영 레벨의 구체성(법적 준비, 시스템 통합, GTM 실행 일정)은 부족**한 상태입니다.

특히:
- **ISMS-P 인증 타임라인** 부족 → 법적 위험
- **학사시스템 자동 연동** Phase 2 미루어짐 → "10배 편리함" 미달성
- **행정사 파트너십** 불명시 → 신뢰도 저하
- **W0~W6 구체적 일정** 부재 → 실행 계획 혼동

이 문제들을 W0 사전 준비 단계에서 해결하면, Phase 1 목표(아산권 3개 대학, 1,500명, NPS 40+) 달성 가능성이 높아질 것으로 판단됩니다.
