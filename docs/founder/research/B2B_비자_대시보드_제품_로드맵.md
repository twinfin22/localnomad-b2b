# 로컬노마드 B2B 비자 관리 플랫폼 — 제품 개발 로드맵

> **버전**: v3.1 | **작성일**: 2026-02-13
> **비전**: AI-Native 외국인 체류 관리 OS — 하이어비자보다 저렴하고, 압도적으로 좋은 UX
> **첫 번째 시장**: 지방대 국제교류처 (하이어비자 미진입) → 수도권 대학 → 기업 HR → B2C 연결
> **개발 방식**: AI 에이전트 기반 초고속 개발. 기존 1개월 = 1주일.
> **플랫폼**: Windows 데스크톱 퍼스트 + 모바일(웹) 호환

## 1. 전략적 맥락

### 1.1 시장 기회

한국의 외국인 체류 관리 시장은 세 가지 구조적 힘이 교차하는 지점에 있다.

**수요 측**: 인구 절벽으로 유학생 유치와 전문 인력(E-7) 고용이 생존 전략으로 격상되었다. 2025년 기준 체류 외국인 250만 명 시대에 진입했으며, 법무부는 E-7-4 쿼터 확대, F-1-D(디지털 노마드) 비자 신설 등 적극적 개방 정책을 추진 중이다. 교육부의 IEQAS(교육국제화역량인증제)는 불법체류율 1% 미만을 핵심 지표로 요구하며, 인증 탈락 시 유학생 유치 자체가 불가능해지므로 대학에게는 생존 이슈이다.

**공급 측**: 대학 국제교류처는 FIMS의 일방향 보고 구조와 LMS 미연동으로 이중 데이터 관리에 시달리고 있다. 업무의 70% 이상이 비자 연장 서류 안내 등 반복적 CS에 소요되며, 불법체류율 1% 미만이라는 교육부 지표 압박 아래 사후약방문식 대응을 반복하고 있다.

**기술 측**: 생성형 AI, OCR, RAG 기술의 성숙으로 서류 자동화와 자연어 상담이 실용 단계에 진입했다. 2025년 모바일 외국인등록증 도입은 핀테크 연동의 새로운 가능성을 열었다. AI 에이전트를 활용한 개발로 기존 대비 4-5배 빠른 제품 출시가 가능하다.

### 1.2 핵심 전략: 하이어비자 대체

하이어비자가 94개 대학을 확보했지만, 이는 곧 94개 대학이 이 시장에 돈을 쓸 의향이 있다는 검증이기도 하다. 우리의 전략은 명확하다.

**하이어비자의 강점을 그대로 흡수하되, 약점을 파고든다:**

| 영역 | 하이어비자 | 로컬노마드 |
|------|-----------|-----------|
| **가격** | 높음 (대학 예산 부담) | 30-50% 저렴한 가격 정책 |
| **UX** | 1세대 대시보드, 엑셀 뷰어 수준 | 모던 UI/UX, Windows 데스크톱 퍼스트 + 모바일 호환 |
| **AI** | 없음 | 다국어 AI 챗봇, 서류 자동화 |
| **알림** | 만료일 기반 단순 알림 | 팩트 기반 다단계 알림 (만료일 + 학적 변동 + 보험 만료 등) |
| **데이터 인수인계** | 3년 보관, 이메일 인증 | 3년+ 보관, 이메일 인증 + 원클릭 마이그레이션 도구 |
| **FIMS 연동** | 수기 이중 입력 | FIMS 양식 호환 내보내기 (파일럿 대학 양식 확보 기반) |
| **CS** | 5개 국어 인간 CS | 6개국어 24/7 AI 챗봇 + 인간 에스컬레이션 |

**진입 순서**: 하이어비자가 미진입한 지방 중위권 대학 → 가격/UX 우위 입증 → 하이어비자 기존 고객 계약 갱신 시점에 전환 유도

### 1.3 규제 안전 구조 (행정사법 대응)

로드맵 전체를 관통하는 법적 원칙:

- **기술 법인과 행정사 사무소의 분리**: 하이어비자의 이원화 모델을 벤치마킹. 플랫폼은 SaaS 도구를 제공하고, 실제 서류 제출은 제휴 행정사가 수행
- **SaaS Defense(삼쩜삼 불기소 논리)**: AI가 생성하는 모든 서류는 "작성 보조 도구"로 포지셔닝. 최종 제출 주체는 대학 담당자 또는 제휴 행정사
- **수익 구조**: 건당 수수료(알선) 모델 절대 금지. 월 정액 구독료(SaaS 이용료) 기반
- **대학 행정 보조 수탁 업체 포지셔닝**: 대학은 출입국관리법상 유학생 비자 단체 접수 권한을 보유한다. 로컬노마드는 대학이 이 권한을 행사하는 데 필요한 "행정 보조 도구"를 제공하는 수탁 업체로 포지셔닝한다. 이는 하이어비자가 "대학 행정 시스템의 일부"로 자리잡은 것과 동일한 법적 구조이며, 행정사법의 적용 범위에서 명확히 벗어나는 포지션이다.
- **결과 보장 표현 금지**: "허가율 보장", "과태료 제로 보장" 등 결과를 보장하는 마케팅 문구 사용하지 않음. 법적 분쟁 소지가 크고, 결과는 출입국 당국의 재량에 달린 사항

### 1.4 개인정보보호 설계 (Privacy by Design)

대학/기업이 SaaS를 도입할 때 가장 큰 허들 중 하나는 개인정보보호법 준수 여부이다. 특히 외국인 체류 관리 데이터는 고유식별정보(여권번호, 외국인등록번호)를 포함하므로 일반 SaaS보다 엄격한 기준이 적용된다.

**A. 법적 관계 정리**
- 대학/기업 = 개인정보 처리자 (Controller)
- 로컬노마드 = 개인정보 수탁자 (Processor)
- 반드시 "개인정보 처리 위탁 계약서"를 체결해야 하며, 대학이 자체 개인정보처리방침에 로컬노마드를 수탁업체로 고지해야 함
- 위탁 계약에 재위탁 금지, 목적 외 이용 금지, 계약 종료 시 파기 의무 등 명시

**B. 고유식별정보 처리 원칙**
- 여권번호, 외국인등록번호 = 개인정보보호법 제24조 "고유식별정보"에 해당
- 처리 근거: 대학의 출입국관리법상 유학생 관리 의무 수행을 위한 위탁 처리
- 별도 동의: 서비스 이용 약관과 분리된 별도 동의서 필수 (동의 안 받으면 불법)
- 암호화: AES-256 암호화 저장 + TLS 1.3 전송 암호화
- 접근 제한: 고유식별정보 열람/조회 시 로그 기록 + 최소 권한 원칙(Need-to-know)

**C. 데이터 흐름과 보안 구조**
- 데이터 입력: 담당자가 엑셀/CSV를 대시보드에 업로드 (학사 시스템과 직접 연동 없음 → 보안 위험 최소화)
- 데이터 저장: 국내 리전 클라우드(AWS 서울/NCloud) + 암호화 + 접근 통제
- 데이터 출력: FIMS 양식 호환 내보내기, 통계 보고서 → 담당자만 다운로드 가능
- 데이터 파기: 보존 기간(3년) 만료 또는 계약 해지 후 6개월 경과 시 복구 불가능한 방식으로 파기
- AI 처리: 챗봇/서류 초안 생성 시 고유식별정보는 마스킹 처리 후 LLM에 전달. 원본 데이터가 외부 AI 서비스로 전송되지 않도록 설계.

**D. 대학 도입 시 필요한 절차**
1. 로컬노마드-대학 간 "개인정보 처리 위탁 계약" 체결
2. 대학이 유학생 대상 "개인정보 제3자 제공/위탁 동의서" 수집 (입학 시 or 서비스 도입 시)
3. 대학 개인정보처리방침에 수탁업체로 로컬노마드 기재
4. ISMS 인증 취득 후 대학 정보보안 심의 통과 (Phase 2 목표)

**E. FIMS 데이터 복사-붙여넣기의 법적 안전성**
- 우리가 하는 것: 대학이 이미 보유한 데이터를 FIMS 양식에 맞게 "정리"해주는 것
- 이는 행정 대행(행정사법 적용)이 아닌, 데이터 포맷 변환 도구(SaaS) 영역
- 비유: 세무사가 세금 신고서를 작성해주고, 납세자가 직접 홈택스에 제출하는 구조와 동일
- 주의: "FIMS 공식 연동" 또는 법무부 협력을 암시하는 표현 금지. "FIMS 양식 호환 내보내기"로 표현

## 2. 제품 비전 — AI-Native 설계 원칙

### "AI가 기본값이다"

전통적 SaaS가 사람이 일하고 소프트웨어가 보조하는 구조라면, AI-Native 플랫폼은 AI가 반복 업무를 처리하고 사람이 판단과 승인에 집중하는 구조다. 모든 기능을 다음 3계층으로 설계한다.

```
┌─────────────────────────────────────────────────┐
│  Layer 3: Human-in-the-Loop (판단/승인)          │
│  담당자가 AI 결과를 검토하고 최종 승인            │
├─────────────────────────────────────────────────┤
│  Layer 2: AI Automation (반복 업무 자동화)        │
│  서류 초안 작성, 팩트 기반 알림 발송,              │
│  다국어 상담, 데이터 정리/변환                     │
├─────────────────────────────────────────────────┤
│  Layer 1: Data Infrastructure (데이터 기반)        │
│  엑셀/CSV 임포트 + AI 컬럼 매핑, OCR, 문서 파싱,  │
│  출입국 규제 DB, 서류 템플릿 DB                    │
└─────────────────────────────────────────────────┘
```

### AI 자동화의 5가지 적용 영역

1. **팩트 기반 알림 시스템**: 비자 만료일, 학적 변동(휴학/제적), 보험 만료, 체류지 변경 신고 기한 등 **확인된 사실**에 기반한 자동 알림. 예측이나 확률 판단은 하지 않음.
2. **AI 서류 자동 생성**: 비자 연장 신청서, 체류자격변경 서류를 학생 데이터 기반으로 자동 초안 작성. 최종 검토는 반드시 담당자 또는 행정사.
3. **AI 다국어 상담 봇**: 베트남어, 우즈베크어, 몽골어 등 주요 유학생 출신국 언어로 24/7 비자 FAQ 상담. 개별 사안에 대한 법적 판단은 하지 않고, 일반적 절차 안내에 한정.
4. **AI 감사 대비(Audit Trail)**: 모든 담당자 행동을 로깅하여 업무 이력 추적 가능
5. **AI 고용사유서 초안 작성**: JD + 이력서 → RAG 기반 E-7 고용사유서 초안 생성 (기업 HR 단계). 반드시 행정사 최종 감수 필수.

## 3. 18주 로드맵 개요

> AI 에이전트 기반 개발로 기존 18개월 로드맵을 18주로 압축한다. 기존 "1개월 = 1주일" 기준.

```
Week    1────3────6────9────12────15────18+
        │         │         │          │
Phase 1 ▓▓▓▓▓▓▓▓▓│         │          │    대학 MVP + 파일럿
        │    Phase 2        │          │
        │    ▓▓▓▓▓▓▓▓▓▓▓▓▓▓│          │    대학 확장 + 기업 진입
        │         │    Phase 3         │
        │         │    ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓│    생태계 + B2C 연결
```

> Phase들은 의도적으로 오버랩된다. Phase 1의 대학 스케일링이 진행되는 동안 Phase 2의 기업 기능 개발이 병행되며, Phase 2의 기업 고객 안정화와 Phase 3의 생태계 구축이 동시에 진행된다.

| Phase | 기간 | 테마 | 핵심 지표 |
|-------|------|------|-----------|
| **Phase 1** | Week 1-6 | 대학 MVP + 지방대 파일럿 | 파일럿 대학 5개, MAU 500 |
| **Phase 2** | Week 4-12 | 대학 확장 + 기업 HR 진입 (Phase 1 스케일링과 병행) | 대학 30개, 기업 50개, MRR 2,000만원 |
| **Phase 3** | Week 10-18+ | 생태계 구축 + B2C 연결 (Phase 2 안정화와 병행) | 대학 100개, 기업 200개, B2C 연동 |

## 4. Phase 1: 대학 MVP + 지방대 파일럿 (Week 1-6)

### 4.1 목표

하이어비자가 하는 것을 더 저렴하게, 더 좋은 UX로 제공한다. 동시에 AI 자동화로 하이어비자가 못하는 것을 추가한다.

- **타겟**: 하이어비자 미진입 지방 중위권 대학 5개 (IEQAS 인증 유지를 위해 불법체류율 1% 미만 관리 압박이 높은 대학 우선)
- **핵심 가치 제안**: "IEQAS 인증 유지의 핵심 — 하이어비자 절반 가격에, 더 좋은 UX와 AI 자동화까지"
- **차별화 무기**: FIMS 양식 호환 내보내기, 6개국어 AI 챗봇, Windows 데스크톱 퍼스트 모던 UI
- **전제 조건**: 파일럿 대학 1곳 이상의 실무 협력 확보 (FIMS 입력 양식, 정기보고 템플릿, 실무 엑셀 양식 수집)

### 4.2 핵심 기능 (MVP Scope)

#### Feature 1: 유학생 통합 관리 대시보드 ⭐ Core

하이어비자의 핵심 기능을 모던 UX로 재구현 + 알파 기능 추가.

**기본 기능 (하이어비자 패리티)**:
- 학생 프로필: 여권 정보, 비자 유형/만료일, 학적 상태, 출석률, 보험 가입 여부, 체류지 정보, 시간제취업 허가 여부/만료일
- 트래픽 라이트 상태: 🟢 정상 / 🟡 주의(만료 60일 이내) / 🔴 긴급(만료 30일 이내) / ⚫ 연락두절
- 비자 만료 캘린더: 월별/주별 뷰로 연장 대상자 한눈에 파악
- 서류 요청 자동화: 비자 연장 시기 도래 시, 학생에게 필요 서류 목록을 다국어로 자동 발송
- Audit Trail: 담당자 행동 로그 (알림 확인 시점, 서류 요청 시점 등)

**차별화 기능**:
- **FIMS 양식 호환 내보내기**: 파일럿 대학에서 확보한 실제 FIMS 입력 양식/엑셀 템플릿을 기반으로, 대시보드 데이터를 FIMS에 복사-붙여넣기 가능한 형태로 자동 변환. FIMS에 공개 API가 없으므로 직접 연동은 불가하며, "데이터 포맷 변환 도구"로 포지셔닝. 양식은 파일럿 대학 실무자에게서 직접 수집하여 확보. 법무부 양식 변경 시 템플릿 업데이트 필요 (유지보수 항목).
- **Windows 데스크톱 퍼스트 UI**: B2B 담당자는 업무를 Windows PC에서 수행. 데스크톱 해상도에 최적화된 넓은 테이블 뷰, 다중 탭, 키보드 단축키 지원. 모바일 웹도 호환되어 이동 중 긴급 알림 확인 가능하나, 핵심 작업 흐름은 데스크톱 기준 설계.
- **팩트 기반 다단계 알림**: 비자 만료일뿐 아니라 학적 변동(휴학/제적), 건강보험 만료, 체류지 변경 신고 기한(14일), 변동신고 기한(15일), 정기보고 마감(연 4회) 등 확인된 사실에 기반한 종합 알림.
- **FIMS 변동신고 워크플로우**: 엑셀 임포트 시 이전 데이터와 비교하여 학적 변동(휴학/제적/자퇴/졸업/미등록 등) 자동 감지 → FIMS 신고 항목에 자동 분류 → 15일 기한 카운트다운 → 신고 데이터 사전 준비 → 담당자가 FIMS에 직접 입력. 신고 완료 여부 체크리스트로 미신고건 추적.
- **정기 현황 보고서 자동 생성**: 2월/5월/8월/11월 연 4회 정기보고 템플릿을 파일럿 대학에서 확보 후, 대시보드 데이터로부터 교육과정별/유학중단사유별 통계를 자동 생성. 보고 시기 1주 전 리마인더 + 데이터 누락 항목 자동 검출.
- **단체 접수 데이터 일괄 준비**: 대학은 출입국관리법상 유학생 비자 단체 접수 권한을 보유. 비자 연장 대상 학생 일괄 선택 → 단체 접수에 필요한 서류 목록 자동 생성 → 학생별 구비 서류 체크리스트 → 미비 서류 자동 알림 → 단체 접수용 통합 데이터 내보내기. 개학기 비자 연장 집중 시기에 담당자 업무량을 대폭 절감.
- **출석률 기반 리스크 조기 감지**: 출석률 데이터를 임포트하여, 출석률 저조(70% 미만) 학생을 자동 식별. 출석률 미달 → 학사 경고 → 제적 → 불법체류로 이어지는 경로를 사전에 차단. IEQAS 인증의 불법체류율 1% 미만 유지에 직결되는 핵심 지표.

**기술 스택**:
- Frontend: Next.js + Tailwind (Windows 데스크톱 퍼스트, 반응형으로 모바일 호환)
- Backend: Node.js + PostgreSQL
- 데이터 입력: 엑셀/CSV 드래그앤드롭 업로드 + AI 자동 컬럼 매핑 + 수동 입력 (학사 시스템과 직접 API 연동 없음 → 보안 위험 최소화)
- 알림 (담당자용): 이메일 + 대시보드 인앱 알림 + 카카오 알림톡
- 알림 (학생용): 카카오 알림톡 + SMS + 이메일 (체류지 변경 14일 기한 리마인더, 비자 만료 사전 알림, 시간제취업 허가 만료 알림 등을 학생 모국어로 발송)

#### Feature 2: AI 다국어 유학생 상담 봇

**문제**: 업무의 70%가 반복적 CS. 다국어 지원 인력 부족. 하이어비자는 인간 CS에 의존하여 비용이 높음.

**솔루션**:
- 비자 연장 절차, 구비 서류, 체류지 변경 신고 방법 등 FAQ 기반 AI 챗봇
- 지원 언어: 한국어, 영어, 중국어, 베트남어, 우즈베크어, 몽골어 (6개국어)
- RAG 아키텍처: 출입국관리법 + 대학별 내규 + FIMS 가이드라인을 벡터 DB로 구축
- **범위 제한**: 일반적 절차 안내에 한정. 개별 사안에 대한 법적 판단이나 허가 가능성 언급은 하지 않음
- 에스컬레이션: AI가 답변 범위 밖이라 판단 시 → 담당자에게 자동 티켓 생성

**기대 효과**: 담당자 CS 업무량 60-70% 감소

#### Feature 3: 데이터 보존 및 인수인계 시스템 ⭐ 신뢰 확보 핵심

하이어비자 전환 시 가장 큰 허들은 "기존 데이터가 날아가면 어떡하나"라는 불안이다. 이를 정면으로 해결한다.

**A. 범용 데이터 마이그레이션 도구**

하이어비자를 쓰던 대학만이 아니라, 아무 시스템도 없이 엑셀/수기로 관리하던 대학도 온보딩할 수 있어야 한다. 대학마다 관리 양식이 제각각이므로, AI 기반 유연한 매핑이 핵심이다.

- **하이어비자 전환**: 하이어비자에서 내보낸 엑셀/CSV 파일 → AI가 필드 구조 자동 분석 → 로컬노마드 데이터 모델에 매핑 → 원클릭 임포트
- **수기/엑셀 관리 대학 전환**: 대학마다 다른 엑셀 양식(컬럼명, 순서, 시트 구조 등)을 AI가 자동 인식. "이름/성명/Name", "여권번호/Passport No." 등 다양한 표기를 자동 매핑. 매핑 결과를 담당자에게 확인용으로 보여준 뒤 최종 임포트.
- **FIMS 내보내기 데이터 역임포트**: 일부 대학은 FIMS에 입력하기 위해 정리해둔 엑셀이 가장 정리된 데이터. 이 엑셀도 임포트 가능.
- **복수 파일 병합**: 학기별, 학과별로 분산된 여러 엑셀 파일을 한 번에 업로드 → AI가 중복 학생 감지 + 최신 데이터 우선 병합
- **마이그레이션 검증 리포트**: 임포트 후 누락 필드, 형식 오류, 중복 데이터를 자동 검출하여 보고. 담당자가 건건이 확인/수정 가능.

**B. 인수인계 안정성**
- **데이터 보존**: 최소 3년 의무 보관. 계약 해지 후에도 6개월간 데이터 열람 가능.
- **접근 제어**: 학교 이메일 인증 기반 접근 + 역할별 권한 관리(RBAC)
- **담당자 교체 프로토콜**: 기존 담당자 → 신규 담당자 권한 이관 워크플로우. 모든 히스토리(알림 이력, 조치 이력, 서류 요청 이력) 완전 보존.
- **인수인계 대시보드**: 신규 담당자가 부임 즉시 "현재 상황 브리핑"을 볼 수 있는 요약 뷰. 긴급 조치 필요 학생, 진행 중인 비자 건, 최근 변동 사항을 한눈에 파악.
- **활동 로그 영구 보관**: 누가, 언제, 무엇을 했는지 전체 감사 추적 가능. 출입국 실태조사 대비.

**C. 데이터 이식성(Portability) 보장**
- 언제든지 전체 데이터를 표준 형식(CSV/JSON)으로 내보내기 가능
- 로컬노마드 해지 시에도 데이터 반출 무료 지원
- 이는 대학이 "플랫폼 락인" 우려 없이 도입할 수 있게 하는 신뢰 장치

## 5. Phase 2: 대학 확장 + 기업 HR 진입 (Week 4-12)

### 5.1 목표

지방대 파일럿 성공 → 수도권 대학으로 확장 + 하이어비자 기존 고객 전환 시작. 동시에 기업 HR 시장에 진입하고 행정사 파트너 네트워크를 구축한다.

- **기업**: 50개 파일럿 (외국인 개발자 5명 이상 고용 IT 스타트업)
- **행정사**: 파트너 행정사 20명 온보딩

### 5.2 대학 기능 고도화

#### Feature 4: 데이터 임포트 고도화 + 학사 시스템 연동 탐색

**Phase 1 기반 고도화 (확정)**:
- 정기 일괄 임포트 자동화: 담당자가 매주/매월 학사 시스템에서 뽑은 엑셀을 업로드하면, 이전 임포트와 자동 비교하여 변동 사항(휴학/제적/성적 변동/출석률 변화) 감지
- 출석률 임포트 및 트렌드 분석: LMS/학사 시스템에서 추출한 출석률 데이터 임포트 → 학생별 출석률 추이 시각화 → 하락 추세 학생 조기 경고
- 성적 미달자 자동 분류: D-2 비자 연장 요건(평균 C학점) 미달 학생 자동 식별 및 리스트 생성
- 휴학/제적 감지 시 → 비자 상태에 미치는 영향 안내 → FIMS 변동신고 워크플로우 자동 트리거

**학사 시스템 직접 연동 (탐색 — 대학 정보보안 심의 통과 전제)**:
- 대학 학사 DB는 개인정보보호법상 민감 시스템이므로, 외부 SaaS와의 API 연동은 대학 정보보안 심의를 통과해야 함
- ISMS 인증 취득 후 (Phase 2 후반) 신뢰 확보된 대학부터 SFTP 또는 암호화 배치 파일 전송 방식으로 시범 연동 시도
- 직접 연동이 불가한 대학은 기존 엑셀 임포트 워크플로우를 계속 지원

#### Feature 5: AI 서류 초안 생성 (유학생)

- 비자 연장 신청서, 체류자격변경 허가신청서 초안 자동 작성
- 학생 프로필 데이터 → 출입국 양식에 맞게 자동 매핑
- **워크플로우**: AI 초안 → 담당자 검토/수정 → 필요 시 제휴 행정사 확인 → 완성
- 법적 안전장치: 초안은 "작성 보조 도구" 산출물. 최종 제출 주체는 담당자 또는 행정사.

### 5.3 기업 HR 신규 기능

#### Feature 6: E-7 컴플라이언스 대시보드

**기업 HR의 핵심 관리 업무를 대시보드화**:

**A. 쿼터 모니터링**
- 고용보험 가입자 명부(엑셀) 업로드 → 내국인/외국인 비율 자동 계산
- 현재 비율 시각화 + 시뮬레이터: "외국인 1명 추가 고용 시 / 내국인 2명 퇴사 시 비율 변화"

**B. GNI 임금 요건 추적**
- 전년도 GNI 발표 시 → 현재 외국인 직원 연봉을 기준과 대조하여 현황 표시
- 기본급 외 수당 포함/제외 시뮬레이션 기능

**C. 신고 기한 추적**
- 고용/해고/퇴직/계약 변경 이벤트 입력 시 → 15일 카운트다운 시작
- 하이코리아 전자신고 양식 자동 생성 (담당자가 직접 제출)

#### Feature 7: AI 고용사유서 초안 작성기

**문제**: E-7 비자 발급 성패의 핵심. HR 담당자는 작성 능력 부족, 행정사는 직무 이해 부족.

**솔루션**:
- 입력: 채용 공고(JD) + 지원자 이력서(Resume) + 회사 정보
- 처리: RAG 엔진이 과거 E-7 승인 사례 DB + 직종 코드별 요건을 참조하여 초안 생성
- 출력: 출입국 심사 기준에 맞는 구조와 키워드를 반영한 초안
- **필수 워크플로우**: AI 초안 → HR 담당자 수정 → **제휴 행정사 최종 감수(Review)** → 완성
- 법적 안전장치: "작성 보조 도구"로 포지셔닝. 최종 제출 주체는 행정사.
- **허가율 보장이나 성공률 표시는 하지 않음**. 결과는 출입국 당국의 재량이며, 우리는 서류 품질을 높이는 데 집중.

**기술 상세**:
- 벡터 DB: 과거 승인된 고용사유서 사례를 비식별화하여 참조
- 프롬프트 엔지니어링: 직종별(IT개발자, 바이오연구원, 글로벌마케터 등) 맞춤
- 피드백 루프: 허가/불허 결과를 참조하여 지속적 품질 향상

#### Feature 8: 직종 코드 매칭 도구

- JD 텍스트 분석 → 한국표준직업분류(KSCO) 코드 후보 제시
- E-7 허용 직종 코드와 대조 + 자격요건(학력, 경력) 체크리스트 제공
- 담당자가 최종 판단하여 선택하는 구조 (AI가 단정하지 않음)

### 6.2 소프트 랜딩 패키지 (B2B → B2C 브릿지)

#### Feature 10: 정착 지원 마켓플레이스

기업이 외국인 직원을 채용하면, 그 직원은 자동으로 B2C 정착 서비스에 연결된다.

**A. ARC-Ready 하우징**
- 외국인등록증 없이 여권만으로 계약 가능한 보증금 제로 단기 임대 매칭
- 제휴: 엔코스테이, 맹그로브 등 코리빙 업체 또는 직영 마스터 리스 매물
- 임대차 계약서 → "거주숙소제공확인서" 자동 변환 (비자 서류 첨부용)

**B. 핀테크 패스**
- 모바일 외국인등록증 API 연동
- 프로세스: 입국 → 앱 설치 → 모바일 ARC 발급 → 제휴 은행 비대면 계좌 개설 → 체크카드 수령

**C. 생활 인프라 온보딩**
- 휴대폰 개통, 교통카드, 건강보험 가입 가이드 → AI 챗봇 + 파트너 연결

### 7.1 시스템 구성

```

│Service │Engine  │Engine  │Service │ Hub         │
├────────┴────────┴────────┴────────┴─────────────┤
│              AI/ML Layer                         │
│  LLM (Claude API) │ RAG Pipeline │ OCR Engine   │
│  Vector DB        │ NLP Pipeline │              │
├─────────────────────────────────────────────────┤
│              Data Layer                          │
│  PostgreSQL │ Redis │ S3 (문서) │ Pinecone (벡터)│
```

### 7.2 AI 스택 상세

| 컴포넌트 | 기술 | 용도 |
|----------|------|------|
| LLM | Claude API (Anthropic) | 고용사유서 초안, 상담 봇, 서류 초안 |
| RAG | LangChain + Pinecone | 규제 DB, 서류 템플릿 검색 증강 생성 |
| OCR | Tesseract + Custom Model | 여권, 외국인등록증, 학위증명서 스캔 |
| NLP | Multilingual BERT | 다국어 의도 분류 |
| 규제 모니터링 | Custom Scraper + LLM | 법무부 고시 변경 감지 및 요약 |

### 7.3 보안 및 데이터 보존 설계

**보안 및 개인정보보호** (상세 내용은 1.4 개인정보보호 설계 참조):
- **데이터 분류**: 여권번호, 외국인등록번호 → Level 4 (최고 등급) → AES-256 암호화 + 접근 로그
- **접근 제어**: 학교 이메일 인증 + RBAC(역할 기반 접근 제어) + 2FA
- **인증 목표**: Phase 2까지 ISMS 인증 취득. Phase 3에서 SOC 2 Type II 추진
- **개인정보 동의**: 고유식별정보 처리 시 서비스 약관과 분리된 별도 동의서 필수
- **법적 관계**: 대학/기업 = 처리자(Controller), 로컬노마드 = 수탁자(Processor). 위탁 계약 필수.
- **AI 처리 시 마스킹**: 고유식별정보는 LLM 전달 전 마스킹. 원본 데이터 외부 전송 방지.
- **데이터 입력 보안**: 학사 시스템 직접 연동 대신 엑셀/CSV 수동 임포트 → 보안 연동 리스크 최소화

**데이터 보존 및 인수인계**:
- **보존 기간**: 최소 3년 의무 보관 + 계약 해지 후 6개월 열람 기간
- **인수인계 프로토콜**: 담당자 교체 시 권한 이관 워크플로우 + 상황 브리핑 뷰
- **데이터 이식성**: 전체 데이터 CSV/JSON 내보내기 상시 가능. 해지 시 데이터 반출 무료.
- **마이그레이션 도구**: 하이어비자/수기 엑셀/FIMS 내보내기 데이터 등 어떤 형태의 기존 데이터든 AI 자동 매핑으로 임포트
- **감사 로그**: 모든 사용자 활동 영구 기록. 출입국 실태조사 대비.

## 11. B2C 연결 전략 (localnomad-website 시너지)

### 현재 B2C (localnomad-website) 작업과의 연결점

| B2B 기능 | B2C 연결점 | 시너지 |
|----------|-----------|--------|
| 대학 유학생 관리 | 유학생이 B2C 앱에서 주거/생활 서비스 이용 | 대학이 "공식 정착 프로그램"으로 채택 |
| 기업 HR 대시보드 | 외국인 직원이 B2C 앱에서 주거/금융 온보딩 | 기업이 복리후생으로 "소프트 랜딩" 제공 |
| AI 비자 상담 봇 | B2C에서 개인 사용자도 동일 봇 이용 | AI 엔진 공유로 개발 비용 최적화 |
| 행정사 CRM | 행정사의 B2C 개인 고객도 플랫폼 유입 | B2B2B → B2B2C 확장 |
| 외국인 프로필 DB | B2C 서비스의 개인화 기반 데이터 | 원타임 데이터 입력으로 모든 서비스 연결 |

### 통합 데이터 모델

```
외국인 프로필 (Unified)
├── 기본 정보: 여권, 국적, 연락처
├── 비자 정보: 체류자격, 만료일, 상태
├── 소속 정보: 대학/기업/프리랜서
├── 주거 정보: 현재 거주지, 계약 상태
├── 금융 정보: 계좌 개설 여부, 보험 가입
└── 활동 로그: 서비스 이용 이력

→ B2B 관리자는 "소속 정보 + 비자 정보" 중심 뷰
→ B2C 사용자는 "주거 + 금융 + 생활" 중심 뷰
→ 같은 데이터, 다른 인터페이스
```

## 12. 장기 비전 (18주 이후)

### 12.1 한국형 Deel/Localyze로의 진화

Phase 3 이후의 확장 방향:

1. **EOR 서비스**: 해외 기업이 한국 인재를 고용할 때, 로컬노마드 법인이 기록상 고용주(EOR) 역할. 파견법 준수 구조 필수.
2. **글로벌 페이롤**: 4대 보험 자동 신고, 연말정산, 퇴직금 계산 엔진 내장
3. **동북아 확장**: 한국 → 대만(골드카드) → 일본으로 확장. "동북아 노마드 패스" 상품

### 12.2 AI 활용 로드맵

| 단계 | 시기 | AI 역할 |

| **Copilot** | Phase 1-2 | AI가 초안/정보 제공, 사람이 검토/판단/승인 |
